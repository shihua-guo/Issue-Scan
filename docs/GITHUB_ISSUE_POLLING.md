# GitHub Issue 轮询任务配置

## 目的
该配置文件用于定义 **GitHub issue 自动扫描的行为规则**，目标是：

- 保证 issue 扫描功能稳定可持续运行
- 尽量把可前置的筛选、去重、过滤逻辑下沉到扫描器中
- 减少 Agent 在被唤醒后的重复扫描成本与 token 消耗
- 让 Agent 的职责聚焦在“理解并处理 issue 原文”，而不是依赖脚本做语义判断

## 当前架构分工

### `Issue-Scan` 的职责
扫描器负责做**前置工作**，包括：

- 扫描哪些仓库
- 只看 open issues
- 只看指定作者
- 过滤无需继续处理的生命周期标签（如 `completed`）
- 根据最近一条非 `[AI]` 评论时间生成去重 marker
- 仅在出现“新 issue / issue 有新用户补充”时才唤醒 Agent
- 预取最小必要上下文：
  - 标题
  - 正文
  - 最新一条非 `[AI]` 回复
- 解析仓库到本地路径映射
- 在安全条件满足时预先执行本地仓库同步

### `GITHUB_ISSUE_POLLING.md` 的职责
本文件负责定义 **Agent 被唤醒后的行为规范**，包括：

- 应该如何回复 issue
- 什么时候只回复，不改代码
- 什么时候允许修改代码 / 提 PR
- 完成后如何收尾

也就是说：

- **扫描器负责“值不值得唤醒”**
- **本文件负责“被唤醒后应该怎么做”**

## 扫描范围
当前默认由扫描器决定扫描哪些仓库；本文件不再把仓库范围硬编码为单一仓库。

如果扫描器扫描多个仓库，则本文件规则对这些仓库中的 issue 一并生效。

## Agent 行为规则

1. **作者限制**
   - 仅处理扫描器已经筛选出来的、作者符合要求的 issue。
   - 不在 Agent 侧重复扩大范围。

2. **AI 回复标记**
   - 每次 AI 在 issue 下回复时，评论正文第一行必须带：
     - `[AI]`
   - 这是当前去重逻辑的一部分，不可省略。

3. **默认处理方式：先回复 issue，不直接改仓库**
   - 默认行为是：
     - 在 issue 下回复分析、结论、追问、进展
   - 不要因为标签名（如 `Idea`、`Plan Mode`）直接决定是否执行。
   - 应优先依据 **issue 原文** 判断用户是在提问、做计划讨论，还是明确要求落地执行。

4. **是否允许改仓库：以 issue 原文为准**
   - 只有当 issue 原文非常明确要求以下行为之一时，才允许进入修改流程：
     - 修改代码
     - 修改文档
     - 提交 PR
     - 修改配置文件
     - 落地实现某个功能或修复
   - 对于纯答疑、说明、分析、排查、计划讨论类 issue：
     - **只在 issue 下回复**
     - **不要直接提交 PR**

5. **不要依赖扫描器分类结果**
   - 扫描器不再基于关键词把 issue 分类为 `plan-only`、`discussion-only`、`implementation-request`。
   - Agent 必须直接依据 issue 原文和最新用户补充来判断处理方式。
   - 不要把标签名或脚本关键词命中结果当作最终语义判断依据。

6. **同一 issue 的后续补充必须继续跟进**
   - 如果用户在同一 issue 下继续补充信息，Agent 必须继续在该 issue 下跟进。
   - 不能因为“之前已回复过一次”就跳过。

7. **完成收尾要求**
   - 当 issue 已明确完成时，Agent 应：
     1. 在 issue 下给出最终回复
     2. 添加 `completed` 标签
     3. 关闭 issue

8. **标签规则**
   - `completed`：视为已完成，不再继续处理
   - 其他标签：仅作参考，不能替代对 issue 原文的理解

## 降低 token 消耗的原则

为减少 token 消耗，优先原则如下：

1. 能在扫描器里过滤的，不留给 Agent
2. 能在扫描器里去重的，不留给 Agent
3. 能在唤醒消息里原样附上的，不让 Agent重复扫描全部上下文
4. Agent 只负责：
   - 理解 issue 内容
   - 给出回复 / 执行明确要求的任务
   - 完成收尾动作

## 建议的后续方向

如果后续还要继续优化 token 消耗，优先考虑：

- 在扫描器中增加更明确的 issue 元信息摘要（非语义裁决）
- 对不同仓库支持独立配置
- 增加更清晰的日志与 dry-run 能力
- 保持“原文透传、Agent 判断”的原则不被新的硬编码分类破坏
